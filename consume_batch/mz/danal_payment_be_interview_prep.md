## 1. 이력서 기반 1분 자기소개 멘트

안녕하세요. 백엔드 개발과 데이터 인프라 운영을 함께 경험해온 개발자입니다.

현재 알바천국에서 검색 서비스 백엔드 개발과 Elasticsearch 인프라 운영을 담당하고 있고, 기존 Mustache Template 기반 검색 구조를 Query DSL 기반으로 전환하면서 검색 로직의 확장성과 유지보수성을 개선한 경험이 있습니다. 또한 Elasticsearch 클러스터를 Windows 환경에서 Linux 환경으로 마이그레이션하고, 모니터링과 알림 체계를 구축해 장애 대응 시간을 줄이고 운영 안정성을 높였습니다.

이전에는 위메프와 큐텐에서 Kafka, Elasticsearch, Redis 기반의 로그/검색 데이터 파이프라인을 운영했고, Qoo10에서는 SQL Server 기반 정산 Stored Procedure 튜닝과 인덱스 설계를 통해 대용량 정산 처리 성능을 개선했습니다.

주요 기술은 Java, Spring 기반 서버 개발과 RDBMS, Elasticsearch, Kafka, Redis 운영 경험이며, 특히 장애가 나면 안 되는 운영 환경에서 데이터 정합성, 성능, 모니터링을 함께 고려해 개선해온 점이 제 강점입니다. 결제개발 BE 직무에서도 승인, 취소, 정산처럼 정확성과 안정성이 중요한 흐름을 이해하고, 운영 가능한 백엔드 시스템을 만드는 데 기여하고 싶습니다.

### 2-5. 결제 도메인에서 자주 나오는 백엔드 면접 질문

결제 도메인은 "돈이 움직이는 상태 전이 시스템"에 가깝기 때문에 다음 주제가 자주 나옵니다.

- 결제 승인, 취소, 환불, 부분취소 흐름을 어떻게 모델링할 것인가
- PG, VAN, 가맹점, 카드사/통신사/은행의 역할 차이는 무엇인가
- 결제 요청 timeout 후 성공 여부를 알 수 없을 때 어떻게 처리할 것인가
- 중복 결제를 어떻게 방지할 것인가
- 멱등성 키를 어떻게 설계하고 저장할 것인가
- DB 트랜잭션과 외부 API 호출을 한 흐름에서 어떻게 다룰 것인가
- 장애 후 재처리, 보상 트랜잭션, 대사 작업을 어떻게 설계할 것인가
- 결제 로그와 감사 추적은 어떤 정보를 남겨야 하는가

Stripe 공식 문서는 네트워크 오류 등으로 응답을 받지 못한 요청을 안전하게 재시도하기 위해 idempotency key를 사용한다고 설명합니다. 특히 POST 요청에 unique key를 붙여 같은 요청이 재시도되어도 중복 생성/중복 과금이 일어나지 않도록 하는 방식이 결제 API 설계에서 중요한 패턴입니다.  
출처: [Stripe Advanced error handling - Idempotency](https://docs.stripe.com/error-low-level?locale=en-GB)

### 2-6. Java/Spring 기반 결제 시스템에서 중요한 기술 주제

- Spring MVC 요청 흐름: Controller, Service, Repository 책임 분리
- Transaction 경계: `@Transactional`, propagation, rollback 정책
- 외부 API 호출과 DB 트랜잭션 분리: 긴 트랜잭션 방지, 상태 저장 후 비동기 확인/보상 처리
- 예외 처리: 비즈니스 예외와 시스템 예외 분리, 공통 응답 포맷, 로그 상관관계 ID
- 동시성 제어: unique constraint, optimistic/pessimistic lock, Redis lock 사용 시 주의점
- 멱등성: 주문번호/결제시도번호/idempotency key 기반 중복 방지
- Batch: Spring Batch chunk, reader/processor/writer, 재시작 가능성, 실패 건 skip/retry
- SQL 튜닝: 실행계획, 인덱스, 조인 순서, 커버링 인덱스, 파티셔닝
- 운영: 모니터링, 알림, 로그 검색, 장애 runbook, 대사/재처리 도구

## 4. 예상 질문별 답변 예시

### A. 자기소개 / 지원동기 / 커리어

#### A-1. 간단히 자기소개해 주세요.

백엔드 개발과 데이터 인프라 운영을 함께 경험해온 개발자입니다. 현재 알바천국에서 검색 서비스 백엔드와 Elasticsearch 클러스터 운영을 담당하고 있고, 검색 쿼리 구조 개선, 템플릿 배포 자동화, Linux 마이그레이션, 모니터링 구축을 수행했습니다. 이전에는 위메프/큐텐에서 Kafka, Elasticsearch, Redis 기반 데이터 파이프라인과 클러스터 운영을 했고, Qoo10에서는 정산 Stored Procedure와 SQL 튜닝을 담당했습니다. 결제개발은 데이터 정합성, 장애 대응, 대용량 처리, 운영 안정성이 모두 중요한 영역이라고 생각해서, 제가 쌓아온 운영형 백엔드 경험을 잘 활용할 수 있다고 봅니다.

#### A-2. 이직을 고려하는 이유는 무엇인가요?

현재 업무에서도 검색 서비스와 데이터 인프라를 안정적으로 운영하며 많이 성장했습니다. 다만 앞으로는 검색 운영을 넘어, 비즈니스 핵심 트랜잭션을 직접 다루는 백엔드 영역에서 더 깊게 성장하고 싶습니다. 결제는 승인, 취소, 정산처럼 데이터 정합성과 장애 대응이 직접적으로 서비스 신뢰도와 연결되는 분야라서 매력을 느꼈습니다. 기존 경험을 버리는 이직이라기보다, 운영 안정성과 대용량 처리 경험을 결제 도메인에 확장하려는 방향입니다.

#### A-3. 왜 다날에 지원했나요?

다날은 휴대폰결제를 시작으로 신용카드, 계좌이체, 상품권, 간편결제, 자동결제까지 다양한 결제 수단을 제공하는 PG/통합결제 플랫폼 기업으로 알고 있습니다. 저는 검색엔진과 데이터 파이프라인을 운영하면서 장애 대응, 모니터링, 정합성 검증의 중요성을 많이 경험했는데, 이런 경험이 결제 플랫폼에서도 중요하다고 생각했습니다. 특히 다날은 오래된 결제 시스템과 신규 결제 서비스를 함께 다루는 회사이기 때문에, 레거시 분석과 구조 개선 경험도 의미 있게 활용할 수 있을 것 같습니다.

#### A-4. 결제개발 BE 직무에 관심을 가진 이유는 무엇인가요?

결제 시스템은 단순히 API를 만드는 수준이 아니라, 요청의 상태를 정확하게 관리하고 실패 상황에서도 데이터가 틀어지지 않게 만드는 일이 핵심이라고 생각합니다. Qoo10에서 정산 SP 튜닝을 하면서 금액 데이터의 일관성과 배치 성능이 얼마나 중요한지 경험했고, 현재 업무에서도 검색 결과 정합성 검증과 장애 대응을 계속 해왔습니다. 이런 경험을 바탕으로 결제 승인, 취소, 환불, 정산처럼 정확성이 중요한 백엔드 시스템을 개발해보고 싶습니다.

#### A-5. 검색엔진/플랫폼 운영 경험을 결제개발에 어떻게 활용할 수 있나요?

검색과 결제는 도메인은 다르지만 운영 관점에서는 공통점이 많습니다. 대량 요청을 안정적으로 처리해야 하고, 장애 발생 시 원인을 빠르게 추적할 수 있어야 하며, 데이터의 누락이나 불일치를 검증해야 합니다. 저는 Elasticsearch 클러스터 이관 시 데이터 손실 없이 마이그레이션하고, 정적/증분 색인 검증을 통해 정합성을 확인했습니다. 결제 시스템에서도 승인/취소/정산 상태를 로그와 DB 기준으로 추적하고, 재처리 가능한 구조를 만드는 데 이 경험을 활용할 수 있다고 생각합니다.

#### A-6. 지금까지 가장 자신 있는 성과는 무엇인가요?

알바천국에서 Elasticsearch 클러스터를 Windows에서 Linux로 마이그레이션한 경험이 가장 기억에 남습니다. 단순 서버 이전이 아니라 템플릿, 색인 데이터, Analyzer, Mapping, 모니터링까지 함께 검증해야 했고, 데이터 누락이나 매핑 충돌이 생기지 않도록 정적/증분 색인을 비교했습니다. 결과적으로 데이터 손실 없이 이관했고, 검색 성능도 약 15% 개선했으며 라이선스 비용도 줄였습니다. 운영 중인 시스템을 안정적으로 개선했다는 점에서 결제 시스템 운영에도 연결되는 경험이라고 봅니다.

#### A-7. Java/Spring 개발자로서 강점과 보완점은 무엇인가요?

강점은 서버 개발을 인프라와 데이터 흐름까지 함께 보고 접근한다는 점입니다. Java/JPA 기반 색인 파이프라인 유지보수, Query DSL 구조 개선, Kafka/Redis/Elasticsearch 운영을 해왔기 때문에 API 코드만이 아니라 병목, 모니터링, 장애 포인트까지 같이 고려합니다. 보완점은 결제 도메인 자체는 아직 직접 개발 경험이 많지 않다는 점입니다. 그래서 면접 전 PG, VAN, 승인/매입/취소/정산 흐름과 Spring Transaction, Batch, 멱등성 설계를 집중적으로 정리하고 있습니다.

### B. Java / Spring / 백엔드 기술

#### B-1. Spring Boot 애플리케이션의 기본 구조를 설명해 주세요.

일반적으로 Controller는 HTTP 요청/응답과 검증을 담당하고, Service는 비즈니스 로직과 트랜잭션 경계를 담당하며, Repository/Mapper는 DB 접근을 담당합니다. 결제 시스템이라면 Controller에서 승인 요청을 받고, Service에서 주문/결제 상태를 검증한 뒤, Repository를 통해 결제 시도 내역과 상태를 저장하는 식으로 나눌 수 있습니다. 중요한 것은 외부 API 호출, DB 저장, 상태 변경의 책임을 섞지 않고 장애 시 추적 가능한 구조로 만드는 것입니다.

#### B-2. `@Transactional`은 어떻게 동작하고, 어떤 경우 rollback되나요?

Spring의 `@Transactional`은 프록시 기반으로 동작하며, 메서드 진입 시 트랜잭션을 시작하고 정상 종료 시 commit, 예외 발생 시 rollback합니다. 기본적으로 unchecked exception은 rollback 대상이고 checked exception은 별도 설정이 없으면 rollback되지 않습니다. 실무에서는 어떤 예외를 비즈니스 실패로 볼지, 어떤 예외를 시스템 실패로 볼지 구분해서 rollback 정책을 명확히 잡는 것이 중요합니다.

#### B-3. 결제 승인 로직에서 DB 트랜잭션 경계는 어디에 잡는 것이 좋을까요?

외부 결제 API 호출을 DB 트랜잭션 안에 오래 묶는 것은 피하는 편이 좋다고 생각합니다. 먼저 결제 시도 상태를 `READY` 또는 `REQUESTED`로 저장하고, 외부 승인 API를 호출한 뒤, 응답 결과에 따라 `APPROVED`나 `FAILED`, `UNKNOWN` 같은 상태로 짧게 업데이트하는 식이 안전합니다. timeout처럼 결과를 모르는 경우에는 별도 조회 API나 대사 배치를 통해 최종 상태를 확인하고 보정할 수 있어야 합니다.

#### B-4. 예외 처리는 어떤 기준으로 설계하나요?

비즈니스 예외와 시스템 예외를 구분합니다. 예를 들어 잔액 부족, 이미 취소된 결제, 중복 요청은 비즈니스 예외로 명확한 코드와 메시지를 내려야 하고, DB 장애나 외부 API timeout은 시스템 예외로 로그와 알림, 재처리 대상이 되어야 합니다. 운영 관점에서는 에러 코드, 주문번호, 결제번호, 외부 거래번호, trace id를 함께 남겨 원인 추적이 가능하게 하는 것이 중요합니다.

#### B-5. JPA와 MyBatis 또는 SQL 직접 사용의 장단점은 무엇인가요?

JPA는 도메인 모델 중심 개발과 변경 감지, 연관관계 관리에 장점이 있지만 복잡한 조회나 대량 배치에서는 실행 SQL을 명확히 확인해야 합니다. MyBatis나 SQL 직접 사용은 복잡한 정산 쿼리, 튜닝이 필요한 집계, 대량 처리에서 제어력이 좋습니다. Qoo10에서 정산 Stored Procedure와 실행계획 튜닝을 했던 경험상, 결제/정산처럼 성능과 정확성이 중요한 영역에서는 ORM을 쓰더라도 핵심 쿼리는 실행계획과 인덱스를 반드시 확인해야 한다고 생각합니다.

#### B-6. API 설계 시 중요하게 보는 기준은 무엇인가요?

요청/응답 스펙의 명확성, 상태 전이의 일관성, 실패 시 재시도 가능성, 로그 추적 가능성을 중요하게 봅니다. 특히 결제 API는 같은 요청이 두 번 들어와도 결과가 예측 가능해야 하므로 idempotency key나 주문번호 기반 중복 방지가 필요합니다. 또한 응답에는 내부 상태와 외부 거래 결과를 혼동하지 않도록, 내부 결제 상태와 외부 승인 결과를 명확히 매핑해야 한다고 생각합니다.

#### B-7. 인증/인가는 어떻게 이해하고 있나요?

인증은 사용자가 누구인지 확인하는 과정이고, 인가는 인증된 사용자가 특정 리소스나 기능에 접근할 권한이 있는지 확인하는 과정입니다. 결제 시스템에서는 사용자 인증뿐 아니라 가맹점 인증, API key/secret 관리, IP 제한, 서명 검증 같은 서버 간 인증도 중요하다고 봅니다. 외부 연동에서는 요청 위변조를 막기 위해 timestamp, nonce, signature 검증 같은 방식도 함께 고려해야 합니다.

#### B-8. 동시에 같은 결제 요청이 두 번 들어오면 어떻게 막을 수 있나요?

우선 주문번호나 결제시도번호에 unique constraint를 걸어 DB 레벨에서 중복을 막습니다. 애플리케이션에서는 idempotency key를 받아 최초 요청의 처리 결과를 저장하고, 동일 키 요청은 새로 처리하지 않고 기존 결과를 반환하도록 설계할 수 있습니다. 매우 짧은 시간에 동시 요청이 들어오는 경우에는 DB unique constraint가 최종 방어선이고, 필요하면 pessimistic lock이나 Redis lock을 사용할 수 있지만 락만 믿기보다는 데이터 모델 자체가 중복을 허용하지 않도록 설계하는 것이 중요합니다.

#### B-9. 배치 처리 경험과 실시간 처리의 차이를 설명해 주세요.

실시간 처리는 요청이 들어온 시점에 즉시 처리하고 응답해야 하므로 지연 시간과 장애 격리가 중요합니다. 배치는 일정 기간의 데이터를 모아 처리하기 때문에 처리량, 재시작 가능성, 실패 건 분리, 정합성 검증이 중요합니다. 저는 정적/증분 색인 파이프라인, 개인정보 자동 파기 스케줄러, Kafka 로그 파이프라인, 정산 SP 튜닝을 경험했습니다. 결제에서는 실시간 승인과 별도로 정산/대사/재처리 배치가 중요하다고 이해하고 있습니다.

### C. 결제 시스템

#### C-1. 결제 승인/취소/환불 흐름을 아는 대로 설명해 주세요.

일반적인 흐름은 가맹점에서 주문이 생성되고, 결제 요청이 PG로 전달되며, PG가 카드사/통신사/은행 등 결제 기관과 승인 처리를 진행합니다. 승인이 성공하면 내부 결제 상태를 승인 완료로 저장하고 가맹점에 결과를 전달합니다. 취소나 환불은 승인된 거래를 기준으로 전체 또는 부분 취소 요청을 보내고, 외부 기관 결과와 내부 상태를 함께 업데이트해야 합니다. 핵심은 각 단계의 상태를 명확히 남기고, 실패나 timeout 시 최종 상태를 조회/대사로 보정할 수 있게 하는 것입니다.

#### C-2. PG, VAN, 가맹점, 카드사의 역할을 설명해 주세요.

가맹점은 상품이나 서비스를 판매하고 결제를 요청하는 주체입니다. PG는 여러 결제수단과 가맹점을 연결해 온라인 결제 승인, 취소, 정산을 중계하는 역할을 합니다. VAN은 오프라인 카드 결제망에서 가맹점과 카드사를 연결하는 승인 중계 역할로 이해하고 있습니다. 카드사는 실제 카드 승인 여부를 판단하고 매입/청구와 연결되는 금융기관입니다. 다날은 PG사로서 가맹점과 여러 결제기관 사이의 안정적인 결제 흐름을 제공하는 역할이라고 이해하고 있습니다.

#### C-3. 결제 시스템에서 트랜잭션 정합성이 중요한 이유는 무엇인가요?

결제는 금액과 사용자 신뢰가 직접 연결되기 때문입니다. DB에는 실패로 남았는데 외부 기관에는 승인된 상태라면 사용자는 결제됐는데 서비스는 받지 못하는 문제가 생길 수 있고, 반대로 내부는 승인인데 외부는 실패면 정산 불일치가 발생합니다. 그래서 내부 거래번호, 외부 거래번호, 상태, 금액, 요청/응답 로그를 기준으로 언제든 최종 상태를 확인하고 보정할 수 있어야 합니다.

#### C-4. 중복 결제를 방지하는 방법은 무엇인가요?

주문번호와 결제시도번호를 분리하고, 결제시도 단위에 unique constraint를 둡니다. 동일 주문에 대해 결제 가능 상태인지 먼저 확인하고, 이미 승인 완료된 주문이면 추가 승인을 막아야 합니다. 외부 API 호출에는 idempotency key를 사용하거나, 외부가 지원하지 않으면 내부에서 요청 키와 처리 결과를 저장해 재시도 시 같은 결과를 반환하도록 설계합니다.

#### C-5. 멱등성 처리를 어떻게 설계하겠습니까?

클라이언트 또는 서버에서 결제 시도별 idempotency key를 생성하고, 요청 payload의 핵심 값과 함께 저장합니다. 최초 요청은 `PROCESSING` 상태로 저장하고 처리가 끝나면 결과 상태와 응답을 저장합니다. 같은 키로 재요청이 오면 payload가 동일한지 검증한 뒤, 이미 완료된 건은 저장된 결과를 반환하고 처리 중인 건은 조회 유도 또는 잠시 후 재시도 응답을 줄 수 있습니다. 저장소는 RDB unique constraint를 기본으로 두고, Redis는 빠른 조회/TTL 보조 용도로 사용할 수 있다고 생각합니다.

#### C-6. 외부 결제 API timeout이 발생했는데 실제 승인 여부를 모르면 어떻게 처리하겠습니까?

바로 실패로 확정하지 않고 `UNKNOWN` 또는 `PENDING_CONFIRM` 같은 중간 상태로 저장하는 것이 안전합니다. 이후 외부 거래 조회 API로 최종 승인 여부를 확인하고, 승인이라면 내부 상태를 승인 완료로 보정하고, 실패라면 실패로 마감합니다. 조회도 실패하면 대사 배치나 운영 알림 대상으로 남겨 수동/자동 재확인이 가능해야 합니다. 이때 같은 요청을 무조건 다시 승인 호출하면 중복 결제 위험이 있으므로, idempotency key나 외부 거래번호 기준으로 재시도해야 합니다.

#### C-7. 결제 취소/환불 실패 시 어떻게 대응하겠습니까?

취소 요청 전 내부 상태를 먼저 검증하고, 취소 시도 이력을 남긴 뒤 외부 취소 API를 호출합니다. 실패가 비즈니스 실패인지, 일시적 시스템 실패인지 구분해야 합니다. 일시 실패라면 재시도 큐나 배치로 재처리하고, 결과를 알 수 없는 timeout이면 외부 거래 조회로 최종 상태를 확인합니다. 부분취소가 있는 경우에는 원 승인 금액, 기존 취소 금액, 잔여 취소 가능 금액을 DB 제약과 트랜잭션으로 함께 검증해야 합니다.

#### C-8. 결제 로그 및 감사 추적에는 어떤 정보를 남겨야 하나요?

내부 주문번호, 결제번호, 결제시도번호, 외부 거래번호, 가맹점 ID, 금액, 결제수단, 요청 시각, 응답 시각, 상태 변경 전후, 에러 코드, trace id를 남겨야 한다고 생각합니다. 민감정보는 마스킹하거나 저장하지 않아야 하며, 원문 요청/응답을 저장해야 한다면 암호화와 접근 권한 관리가 필요합니다. 저는 Kafka/Elasticsearch 기반 통합 로깅 파이프라인과 Kibana/Grafana 모니터링을 구축한 경험이 있어, 결제 로그도 추적성과 보안성을 함께 고려해 설계할 수 있습니다.

### D. 데이터베이스 / 대용량 처리

#### D-1. 인덱스 설계 시 어떤 기준을 보나요?

쿼리의 where 조건, join 조건, order by/group by, 카디널리티, 조회 빈도와 쓰기 비용을 함께 봅니다. Qoo10에서 정산 SP 튜닝을 할 때 실행계획을 보고 Full Scan, 비효율적인 Join/Subquery, Cursor 구간을 줄였고, 조회 패턴에 맞춰 Non-Clustered/Composite Index를 설계했습니다. 결제 시스템에서는 주문번호, 거래번호, 가맹점 ID, 상태, 거래일시 같은 컬럼이 주요 조회 조건이 될 가능성이 높아 복합 인덱스 설계가 중요할 것 같습니다.

#### D-2. SQL 튜닝을 어떻게 진행했나요?

먼저 느린 쿼리와 실행계획을 확인해 병목이 Full Scan인지, Join 순서인지, 인덱스 미사용인지, 불필요한 Cursor/반복 호출인지 파악합니다. 이후 조건절과 인덱스 구조를 맞추고, 필요한 경우 집합 기반 처리로 바꾸며, 변경 전후 실행 시간과 IO를 비교합니다. Qoo10에서는 정산 Stored Procedure 처리 시간을 수십 초에서 수 초 수준으로 줄인 경험이 있습니다.

#### D-3. Lock과 Deadlock을 설명해 주세요.

Lock은 동시에 같은 데이터를 변경할 때 정합성을 지키기 위한 제어 장치입니다. Deadlock은 두 트랜잭션이 서로 상대가 가진 Lock을 기다리면서 더 이상 진행하지 못하는 상태입니다. 예방하려면 트랜잭션을 짧게 유지하고, 테이블/레코드 접근 순서를 일관되게 가져가며, 적절한 인덱스로 불필요한 범위 Lock을 줄이는 것이 중요합니다. 결제에서는 같은 주문이나 거래를 여러 요청이 동시에 변경하지 않도록 상태 전이와 Lock 범위를 신중히 설계해야 합니다.

#### D-4. 트랜잭션 격리 수준을 설명해 주세요.

격리 수준은 동시에 실행되는 트랜잭션 간 데이터가 어떻게 보이는지를 정하는 기준입니다. Read Uncommitted, Read Committed, Repeatable Read, Serializable 순으로 격리가 강해지고 동시성은 낮아질 수 있습니다. 격리 수준에 따라 dirty read, non-repeatable read, phantom read 문제가 달라집니다. 결제 시스템에서는 무조건 높은 격리 수준을 쓰기보다, unique constraint, 상태 조건부 update, lock을 조합해 필요한 정합성을 확보하는 것이 현실적이라고 생각합니다.

#### D-5. 대용량 데이터 처리 경험을 설명해 주세요.

위메프/큐텐에서 상품 검색 색인 파이프라인을 운영하며 대량 상품 데이터의 Bulk Indexing, Refresh, Alias 전환, 증분 색인 정책을 다뤘습니다. Kafka 기반 통합 로그 파이프라인도 설계해 API 로그를 Kafka로 수집하고 Elasticsearch에 저장해 실시간 분석 환경을 만들었습니다. Qoo10에서는 정산 관련 대용량 조회와 Stored Procedure 튜닝을 했습니다. 이 경험은 결제 정산 배치나 거래 로그 분석에도 연결된다고 생각합니다.

#### D-6. 정산 배치를 설계한다면 어떤 점을 고려하나요?

정산 기준일, 가맹점, 결제수단, 승인/취소/부분취소 상태를 명확히 정의해야 합니다. 대용량 처리를 위해 배치 단위와 chunk 크기, 재시작 지점, 실패 건 분리, 중복 실행 방지, 결과 검증 리포트가 필요합니다. 또한 외부 기관 정산 데이터와 내부 거래 데이터를 대사해 차이를 탐지하고, 차이 건은 재처리 또는 운영 확인 대상으로 분리해야 합니다. 제 정산 SP 튜닝 경험과 배치/파이프라인 운영 경험을 활용할 수 있는 영역이라고 봅니다.

#### D-7. Redis나 Kafka를 결제 시스템에서 어떻게 활용할 수 있을까요?

Redis는 idempotency key의 빠른 조회, 짧은 TTL의 중복 요청 방지, rate limit, 세션/토큰 캐시 등에 활용할 수 있습니다. 다만 결제 정합성의 최종 기준은 RDB가 되어야 하고 Redis는 보조 수단으로 보는 것이 안전합니다. Kafka는 결제 이벤트, 로그, 알림, 대사/정산 후속 처리를 비동기로 분리하는 데 활용할 수 있습니다. 저는 Kafka 토픽/파티션 설계와 Redis Replication/Sentinel, Slowlog/Latency 모니터링 경험이 있습니다.

### E. Elasticsearch / Kafka / Redis / 운영

#### E-1. Elasticsearch 운영 경험을 설명해 주세요.

알바천국과 위메프/큐텐에서 Elasticsearch 기반 검색 시스템을 운영했습니다. Query DSL, Mapping, Analyzer, nori 형태소 분석기, 사용자 사전, 동의어, edge_ngram, 초성/부분일치 전략을 다뤘고, Bulk Indexing과 Alias 전환, 증분 색인 정책도 운영했습니다. 또한 클러스터 메트릭을 수집해 Grafana/Kibana 대시보드를 구성하고 장애 알림을 연동했습니다.

#### E-2. Elasticsearch 장애 대응 경험은 무엇이 있나요?

검색 결과 누락, 매핑 충돌, 색인 지연, 리소스 사용량 증가 같은 이슈를 다뤘습니다. 장애가 발생하면 먼저 Cluster Health, shard 상태, GC, query cache, disk, CPU/memory를 확인하고, 색인 로그와 검색 로그를 함께 확인했습니다. 알바천국에서는 Rust 기반 로그/메트릭 수집과 Telegram 알림을 구축해 장애 대응 시간을 30분에서 10분 수준으로 줄였습니다.

#### E-3. 검색 템플릿 배포 자동화는 왜 만들었고 효과는 무엇이었나요?

기존 Mustache Template 검색 쿼리는 수작업 관리라 배포 충돌, 변경 이력 부재, 책임자 추적 문제가 있었습니다. 그래서 Rust CLI로 Git Repository의 템플릿을 파싱하고 Elasticsearch `_scripts` API로 자동 등록/수정하는 Jenkins 파이프라인을 만들었습니다. 결과적으로 배포 시간이 0~1분 수준으로 줄었고, 개발/운영 브랜치를 분리해 운영 리스크와 충돌을 줄였습니다. 결제 시스템에서도 배포 자동화와 변경 이력 추적은 운영 리스크를 줄이는 중요한 장치라고 생각합니다.

#### E-4. Kafka 기반 로그 파이프라인을 어떻게 설계했나요?

그룹사별로 상이한 로그 구조를 API -> Kafka -> Elasticsearch 구조로 통합했습니다. Kafka 토픽을 그룹사/로그 성격별로 설계하고, Elasticsearch 인덱스와 매핑을 구성한 뒤 Kibana 대시보드로 분석할 수 있게 했습니다. 이후 Kafka + Logstash 구조를 Kafka + Vector 구조로 전환해 메모리 사용량을 줄이고 처리 속도를 높인 경험도 있습니다. 결제에서도 이벤트 로그를 안정적으로 수집하고 재처리 가능한 구조를 만들 때 Kafka 경험이 도움이 될 수 있습니다.

#### E-5. Redis 운영 시 주의한 점은 무엇인가요?

Redis는 빠르지만 싱글 스레드 기반이므로 O(N) 계열 고비용 명령어를 조심해야 합니다. 위메프에서 Redis master-slave 구조와 모니터링을 구축했고, `rename-command`로 위험 명령어를 제한했습니다. Slowlog와 Latency를 수집해 성능 저하를 조기 감지하는 체계도 만들었습니다. 결제 시스템에서 Redis를 쓴다면 락이나 캐시가 장애났을 때도 RDB 정합성이 깨지지 않도록 설계해야 한다고 생각합니다.

#### E-6. 모니터링/알림 시스템을 어떻게 구축했나요?

Metricbeat, Heartbeat, Rust 기반 수집 로직, Elasticsearch, Grafana/Kibana, Telegram 알림을 조합해 Elasticsearch/Kafka/Redis 상태를 모니터링했습니다. CPU, memory, disk, shard, GC, query cache, Kafka 처리량, Redis memory/latency 같은 지표를 수집했고 임계치 기반 알림을 구성했습니다. 결제 시스템에서는 승인 실패율, timeout 비율, 상태 불일치 건수, 외부 API 응답 시간, 정산 배치 실패 건수 같은 지표가 중요할 것 같습니다.

#### E-7. Linux 마이그레이션에서 가장 중요했던 점은 무엇인가요?

데이터 손실 없이 기능 호환성을 검증하는 것이 가장 중요했습니다. Windows 클러스터의 설정, 템플릿, Analyzer, Mapping, 색인 데이터를 분석하고 Linux 클러스터에 이관한 뒤 정적/증분 색인을 비교했습니다. systemd 기반 서비스 관리와 로그 운영 체계도 재구성했습니다. 결제 시스템에서도 인프라 전환이나 레거시 개선 시 기존 결과와 신규 결과의 정합성 검증이 핵심이라고 생각합니다.

### F. 인성 / 협업 / 장애 대응

#### F-1. 장애를 해결했던 경험을 말해 주세요.

Elasticsearch 클러스터 운영 중 검색 장애나 품질 이슈가 발생하면 쿼리, 사전, Analyzer, 색인 상태, 노드 리소스를 함께 확인해 원인을 좁혔습니다. 또한 장애 대응 시간을 줄이기 위해 로그와 메트릭을 자동 수집하고 Telegram 알림을 붙였습니다. 단순히 장애를 한 번 해결하는 데서 끝내지 않고, 같은 문제가 다시 발생했을 때 더 빨리 감지하고 대응할 수 있도록 모니터링 체계를 만든 점이 의미 있었습니다.

#### F-2. 운영 중 실수를 방지하기 위해 어떤 장치를 두나요?

수작업을 줄이고 자동화와 검증 단계를 둡니다. 검색 템플릿 배포도 수작업으로 운영하면 충돌과 누락이 생길 수 있어 Git/Jenkins 기반 자동 배포로 바꾸고, 개발/운영 브랜치를 분리했습니다. 인프라 설치도 Bash Script로 표준화해 서버별 설정 편차를 줄였습니다. 결제 시스템에서도 운영 실수를 막기 위해 배포 이력, 승인 절차, dry-run, 검증 쿼리, rollback 또는 보정 절차가 필요하다고 생각합니다.

#### F-3. 협업 시 갈등이 있었던 경험과 해결 방법은 무엇인가요?

검색 템플릿 관리 방식처럼 여러 개발자가 함께 사용하는 영역에서는 변경 책임과 배포 타이밍이 불명확해 충돌이 생길 수 있습니다. 이때 특정 사람의 실수로 보기보다 프로세스 문제로 보고, Git 기반 이력 관리와 Jenkins 자동 배포 구조를 제안하고 구현했습니다. 결과적으로 누가 언제 무엇을 바꿨는지 추적 가능해졌고, 서비스 개발자는 파라미터만 전달해 검색 기능을 연동할 수 있게 되었습니다.

#### F-4. 모르는 기술을 빠르게 익힌 경험이 있나요?

위메프 검색개발팀 인력 공백 상황에서 Elasticsearch 기반 상품 검색 플랫폼을 단독으로 인계받은 경험이 있습니다. Query DSL, Analyzer, nori 형태소 분석기, 사전, 색인 파이프라인, 클러스터 운영까지 빠르게 파악해야 했습니다. 운영 이슈를 해결하면서 공식 문서와 기존 코드, 로그를 함께 보며 학습했고, 이후 검색 품질과 운영 안정성을 개선했습니다. 결제 도메인도 처음부터 겸손하게 배우되, 운영 로그와 데이터 흐름을 빠르게 파악하는 방식으로 적응하겠습니다.

#### F-5. 주도적으로 개선한 경험을 말해 주세요.

Elasticsearch 검색 템플릿 배포 자동화 시스템을 주도적으로 구축했습니다. 기존에는 수작업 배포라 시간이 오래 걸리고 변경 이력 추적이 어려웠습니다. Rust CLI와 Jenkins Pipeline, Git 브랜치 전략을 도입해 템플릿 등록/수정을 자동화했고, 배포 시간을 90% 이상 단축했습니다. 결제개발에서도 반복적이고 위험한 운영 작업을 자동화하고, 장애 가능성을 줄이는 개선을 주도하고 싶습니다.

#### F-6. 본인이 생각하는 좋은 백엔드 개발자는 어떤 사람인가요?

기능을 구현하는 데서 끝나지 않고, 운영 중 어떤 문제가 생길 수 있는지까지 생각하는 사람이라고 봅니다. 특히 결제처럼 중요한 시스템에서는 정상 케이스보다 실패, 중복, timeout, 재처리, 로그 추적을 더 꼼꼼히 봐야 합니다. 저는 검색과 데이터 인프라를 운영하면서 장애를 빨리 감지하고, 원인을 추적하고, 재발을 줄이는 구조를 만드는 일을 많이 해왔습니다. 그런 관점을 가진 백엔드 개발자로 기여하고 싶습니다.

## 5. 최종 면접 준비 요약

### 5-1. 면접에서 강조해야 할 핵심 강점 3가지

1. 운영 안정성 중심의 백엔드 경험  
   Elasticsearch, Kafka, Redis, Linux 환경을 직접 운영하며 장애 대응, 모니터링, 자동화 경험을 쌓았습니다.

2. 데이터 정합성과 대용량 처리 경험  
   정산 Stored Procedure 튜닝, 대량 색인 파이프라인, Kafka 로그 파이프라인, 개인정보 파기 스케줄러 경험이 결제/정산 시스템과 연결됩니다.

3. 레거시 분석 및 구조 개선 역량  
   Mustache Template -> Query DSL 전환, Windows -> Linux 마이그레이션, Logstash -> Vector 전환처럼 기존 구조를 분석하고 안정적으로 개선한 경험이 있습니다.

### 5-2. 다날 결제개발 BE와 연결되는 포인트

- 다날은 PG/통합결제 플랫폼 기업이고, 결제개발 BE는 승인/취소/정산/대사/운영 안정성이 중요합니다.
- 본인의 Qoo10 정산 SP 튜닝 경험은 결제 정산 플랫폼과 직접적으로 연결됩니다.
- Kafka/Elasticsearch 로그 파이프라인 경험은 결제 로그, 감사 추적, 장애 분석 체계와 연결됩니다.
- Redis/Kafka/DB 운영 경험은 중복 결제 방지, 비동기 처리, 대용량 정산 배치 설계에 연결됩니다.
- Linux 마이그레이션과 모니터링 경험은 결제 시스템의 운영 안정화 역량으로 어필할 수 있습니다.

### 5-3. 면접 전 반드시 복습할 기술 키워드

- Java: Collection, Stream, Exception, Thread 기본
- Spring Boot: MVC 구조, Bean, DI, AOP, Filter/Interceptor
- Transaction: `@Transactional`, propagation, isolation, rollback
- DB: index, execution plan, lock, deadlock, isolation level, unique constraint
- SQL: join, group by, covering index, pagination, batch update
- Spring Batch: Job, Step, Chunk, Reader/Processor/Writer, restart, skip/retry
- 결제: PG, VAN, 가맹점, 카드사, 승인, 취소, 환불, 부분취소, 정산, 대사
- 안정성: idempotency key, retry, timeout, circuit breaker, compensation transaction
- Kafka: topic, partition, consumer group, offset, at-least-once, ordering
- Redis: cache, TTL, distributed lock, replication, sentinel, slowlog
- Elasticsearch: index, shard, replica, mapping, analyzer, alias, bulk indexing
- 운영: monitoring, alerting, trace id, log correlation, runbook

### 5-4. 답변 시 조심해야 할 부분

- "결제 도메인을 잘 안다"고 과장하지 말고, "정산/정합성/운영 경험을 결제 도메인으로 확장하고 있다"고 말하는 편이 자연스럽습니다.
- Java/Spring 경험은 실제 프로젝트에서 어느 정도까지 했는지 솔직하게 말하고, 대신 서버/데이터/운영 전체를 보는 강점을 강조하세요.
- Elasticsearch 경험만 길게 말하면 직무와 멀어질 수 있으니, 항상 "정합성, 장애 대응, 로그 추적, 대용량 처리"로 결제와 연결해 마무리하세요.
- Redis lock, Kafka 비동기 처리 같은 기술을 만능처럼 말하지 말고, 최종 정합성은 RDB 제약과 상태 모델로 지킨다고 답하는 것이 좋습니다.
- timeout 상황에서 무조건 재시도한다고 말하면 중복 결제 위험이 있어 보입니다. 반드시 idempotency key, 상태 조회, 대사/보정 흐름을 함께 말하세요.
