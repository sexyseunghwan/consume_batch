## 1. 이력서 기반 1분 자기소개 멘트

안녕하세요. 백엔드 개발과 데이터 인프라 운영을 함께 경험해온 개발자입니다.

현재 알바천국에서 검색 서비스 백엔드 개발과 Elasticsearch 인프라 운영을 담당하고 있고, 기존 Mustache Template 기반 검색 구조를 Query DSL 기반으로 전환하면서 검색 로직의 확장성과 유지보수성을 개선한 경험이 있습니다. 또한 Elasticsearch 클러스터를 Windows 환경에서 Linux 환경으로 마이그레이션하고, 모니터링과 알림 체계를 구축해 장애 대응 시간을 줄이고 운영 안정성을 높였습니다.

이전에는 위메프와 큐텐에서 Kafka, Elasticsearch, Redis 기반의 로그/검색 데이터 파이프라인을 운영했고, Qoo10에서는 SQL Server 기반 정산 Stored Procedure 튜닝과 인덱스 설계를 통해 대용량 정산 처리 성능을 개선했습니다.

주요 기술은 Java, Spring 기반 서버 개발과 RDBMS, Elasticsearch, Kafka, Redis 운영 경험이며, 특히 장애가 나면 안 되는 운영 환경에서 데이터 정합성, 성능, 모니터링을 함께 고려해 개선해온 점이 제 강점입니다. 결제 Risk Management 개발 직무에서도 이상 거래 탐지, 룰 엔진 운영, 실시간 모니터링처럼 정확성과 안정성이 핵심인 시스템을 개발하고 운영하는 데 기여하고 싶습니다.

### 2-3. 결제 Risk Management 개발 BE 직무에서 중요하게 볼 가능성이 높은 역량

결제 Risk Management 개발 BE 직무는 사기 거래 탐지, 이상 패턴 모니터링, 리스크 룰 엔진 개발, 컴플라이언스 대응 등을 주로 담당하는 포지션입니다. 일반적으로 다음 업무가 포함될 가능성이 높습니다.

- 이상 거래 탐지 시스템(FDS) 개발 및 운영
- 리스크 룰 엔진/스코어링 시스템 개발 및 정책 관리
- 실시간 거래 모니터링 파이프라인 설계 및 운영
- Spring Boot 기반 Risk API/배치 서비스 개발
- 거래 데이터 분석 및 이상 패턴 탐지를 위한 데이터 파이프라인 구축
- 금융 규제 대응(AML, 전자금융거래법 등) 관련 시스템 개발
- RDBMS 기반 리스크 데이터 모델링 및 조회 성능 최적화
- Java/Spring 기반 백엔드 서비스 개발/운영 경험, Kafka/Elasticsearch/Redis 활용 경험

이를 결제 Risk Management BE 1차 면접 관점으로 해석하면, 다음 역량을 중요하게 볼 가능성이 높습니다.

- Java/Spring Boot 기반 API 및 배치 서버 개발 역량
- 실시간 이벤트 스트림(Kafka 등) 기반 거래 모니터링 경험
- 룰 엔진, 스코어링, 임계치 기반 이상 탐지 로직 설계 역량
- Elasticsearch 기반 대용량 거래 로그 검색 및 패턴 분석 경험
- 트랜잭션, 정합성, 예외 처리에 대한 실무 이해
- 장애 상황에서 로그, 모니터링, 재처리로 데이터를 복구하는 운영 역량
- 외부 기관/제휴사 API 연동 시 timeout, retry, 중복 요청을 통제하는 설계 역량

### 2-5. 결제 Risk Management 도메인에서 자주 나오는 백엔드 면접 질문

결제 Risk Management는 "실시간으로 흐르는 거래 데이터에서 이상 패턴을 탐지하고 차단/알림하는 시스템"에 가깝기 때문에 다음 주제가 자주 나옵니다.

- 이상 거래 탐지(FDS)를 어떻게 설계할 것인가
- 룰 엔진 기반 리스크 판정 로직을 어떻게 구현할 것인가
- 실시간 거래 스트림에서 Velocity Check(단시간 다량 결제 등)를 어떻게 처리할 것인가
- 탐지 룰 변경이 잦을 때 배포 없이 정책을 반영할 수 있는 구조는 어떻게 만드는가
- Kafka와 같은 스트림 처리 기반으로 거래 이벤트를 어떻게 소비하고 룰을 적용하는가
- 룰 판정 결과를 RDB와 Elasticsearch에 어떻게 분산 저장/조회하는가
- 이상 탐지 결과에 대해 자동 차단과 수동 검토 케이스를 어떻게 나누는가
- AML(자금세탁방지), 전자금융거래법 대응을 위한 로그/감사 추적을 어떻게 설계하는가

### 2-6. Java/Spring 기반 Risk Management 시스템에서 중요한 기술 주제

- Spring MVC 요청 흐름: Controller, Service, Repository 책임 분리
- Transaction 경계: `@Transactional`, propagation, rollback 정책
- 룰 엔진 설계: 정책/임계치를 DB나 설정에서 동적으로 로드해 적용하는 구조
- 실시간 이벤트 처리: Kafka Consumer 기반 거래 이벤트 소비, 처리 결과 저장
- Velocity Check: Redis를 활용한 단위 시간 내 요청 횟수/금액 집계 및 임계치 초과 탐지
- 동시성 제어: unique constraint, optimistic/pessimistic lock, Redis lock
- 멱등성: 중복 이벤트 처리 방지, 동일 거래번호 중복 룰 판정 방지
- Batch: Spring Batch chunk 기반 대용량 거래 데이터 리스크 분석/집계 배치
- SQL 튜닝: 실행계획, 인덱스, 커버링 인덱스, 대용량 조회 최적화
- Elasticsearch: 거래 로그 검색, 이상 패턴 집계, Kibana 대시보드 운영
- 운영: 모니터링, 알림, 룰 판정 이력 추적, 대사/재처리 도구, runbook

### A. 자기소개 / 지원동기 / 커리어

#### A-1. 간단히 자기소개해 주세요.

백엔드 개발과 데이터 인프라 운영을 함께 경험해온 개발자입니다. 현재 알바천국에서 검색 서비스 백엔드와 Elasticsearch 클러스터 운영을 담당하고 있고, 검색 쿼리 구조 개선, 템플릿 배포 자동화, Linux 마이그레이션, 모니터링 구축을 수행했습니다. 이전에는 위메프/큐텐에서 Kafka, Elasticsearch, Redis 기반 데이터 파이프라인과 클러스터 운영을 했고, Qoo10에서는 정산 Stored Procedure와 SQL 튜닝을 담당했습니다. 결제 Risk Management는 실시간 거래 데이터의 이상 패턴 탐지, 룰 엔진 운영, 감사 로그 관리가 핵심이라고 생각해서, 제가 쌓아온 Kafka 스트리밍, Elasticsearch 기반 로그 분석, 모니터링 경험을 잘 활용할 수 있다고 봅니다.

#### A-2. 이직을 고려하는 이유는 무엇인가요?

현재 업무에서도 검색 서비스와 데이터 인프라를 안정적으로 운영하며 많이 성장했습니다. 다만 앞으로는 실시간 이벤트 스트림 기반으로 이상 패턴을 탐지하고 리스크 판정을 내리는 시스템처럼, 데이터의 흐름 자체가 비즈니스 신뢰성과 직결되는 영역에서 더 깊게 성장하고 싶습니다. 기존 경험을 버리는 이직이라기보다, Kafka 파이프라인, Elasticsearch 로그 분석, 모니터링 체계를 결제 Risk Management 도메인에 확장하려는 방향입니다.

#### A-3. 왜 다날에 지원했나요?

다날은 휴대폰결제를 시작으로 신용카드, 계좌이체, 상품권, 간편결제, 자동결제까지 다양한 결제 수단을 제공하는 PG/통합결제 플랫폼 기업으로 알고 있습니다. 결제 수단이 다양할수록 각 채널별 이상 거래 패턴도 다양하기 때문에, Risk Management 시스템의 중요도와 기술적 복잡성이 높은 환경이라고 생각했습니다. 저는 Kafka 기반 실시간 로그 파이프라인, Elasticsearch 기반 이상 탐지, 모니터링 알림 체계를 구축한 경험이 있어, 이를 결제 Risk Management 시스템에 연결할 수 있다고 판단해 지원했습니다.

#### A-4. 결제 Risk Management 개발 직무에 관심을 가진 이유는 무엇인가요?

결제 Risk Management는 단순히 API를 만드는 수준이 아니라, 실시간으로 흐르는 거래 이벤트에서 이상 패턴을 빠르게 감지하고 자동으로 대응하는 시스템을 만드는 일이 핵심이라고 생각합니다. 저는 Kafka 기반 통합 로그 파이프라인을 설계하면서 실시간 스트림 처리와 Elasticsearch 기반 이상 패턴 집계를 경험했고, Elasticsearch 클러스터 모니터링과 Grafana/Telegram 알림 체계를 직접 구축했습니다. 이런 경험을 바탕으로 거래 이벤트 기반 이상 탐지, 룰 엔진 운영, 감사 로그 관리처럼 정확성과 신뢰성이 중요한 Risk 백엔드 시스템을 개발해보고 싶습니다.

#### A-5. 검색엔진/플랫폼 운영 경험을 Risk Management 개발에 어떻게 활용할 수 있나요?

검색과 Risk Management는 도메인은 다르지만 운영 관점에서는 공통점이 많습니다. 대량 이벤트를 실시간으로 처리해야 하고, 장애 발생 시 원인을 빠르게 추적할 수 있어야 하며, 데이터의 누락이나 패턴 이상을 감지해야 합니다. 저는 Elasticsearch 기반 검색 로그와 Kafka 기반 파이프라인을 통해 데이터 정합성 검증과 이상 감지 모니터링을 구축했습니다. 결제 Risk Management에서도 Kafka로 유입되는 거래 이벤트를 소비해 룰을 적용하고, Elasticsearch로 이상 패턴을 집계하고 시각화하는 구조를 만드는 데 이 경험을 직접 활용할 수 있다고 생각합니다.

#### A-6. 지금까지 가장 자신 있는 성과는 무엇인가요?

알바천국에서 Elasticsearch 클러스터를 Windows에서 Linux로 마이그레이션한 경험이 가장 기억에 남습니다. 단순 서버 이전이 아니라 템플릿, 색인 데이터, Analyzer, Mapping, 모니터링까지 함께 검증해야 했고, 데이터 누락이나 매핑 충돌이 생기지 않도록 정적/증분 색인을 비교했습니다. 결과적으로 데이터 손실 없이 이관했고, 검색 성능도 약 15% 개선했으며 라이선스 비용도 줄였습니다. Risk Management 시스템에서도 거래 로그 인프라를 안정적으로 전환하거나 개선할 때 이런 검증 방식이 그대로 적용된다고 봅니다.

#### A-7. Java/Spring 개발자로서 강점과 보완점은 무엇인가요?

강점은 서버 개발을 인프라와 데이터 흐름 전체로 보고 접근한다는 점입니다. API 코드만이 아니라 Kafka Consumer 처리 지연, Redis 캐시 오작동, Elasticsearch 색인 지연, 모니터링 알림 누락처럼 운영 중에 나타나는 병목과 장애 포인트까지 함께 고려합니다. Risk Management 관점에서 이 강점이 중요한 이유는, 이상 탐지 시스템은 단순히 API가 응답을 주는 것보다 "거래 이벤트가 Consumer에 도달하고 룰이 정확하게 적용되고 판정 이력이 빠짐없이 남는 전체 흐름"이 중요하기 때문입니다. 보완점은 결제 Risk Management 도메인 자체의 직접 개발 경험이 아직 많지 않다는 점입니다. 그래서 면접 전 룰 엔진 설계, Velocity Check, 이상 탐지 로직, AML/감사 추적 요건, Spring Transaction/Batch, 멱등성 설계를 집중적으로 정리하고 있습니다.

### B. Java / Spring / 백엔드 기술

#### B-1. Spring Boot 애플리케이션의 기본 구조를 설명해 주세요.

일반적으로 Controller는 HTTP 요청/응답과 검증을 담당하고, Service는 비즈니스 로직과 트랜잭션 경계를 담당하며, Repository/Mapper는 DB 접근을 담당합니다. Risk Management 시스템이라면 Controller에서 리스크 판정 요청을 받고, Service에서 룰 엔진을 호출해 판정을 내린 뒤, Repository를 통해 판정 이력과 거래 상태를 저장하는 식으로 나눌 수 있습니다. 중요한 것은 룰 판정, 이벤트 저장, 외부 차단 API 호출의 책임을 섞지 않고 장애 시 추적 가능한 구조로 만드는 것입니다.

#### B-2. `@Transactional`은 어떻게 동작하고, 어떤 경우 rollback되나요?

Spring의 `@Transactional`은 프록시 기반으로 동작하며, 메서드 진입 시 트랜잭션을 시작하고 정상 종료 시 commit, 예외 발생 시 rollback합니다. 기본적으로 unchecked exception은 rollback 대상이고 checked exception은 별도 설정이 없으면 rollback되지 않습니다. Risk Management 관점에서 중요한 건 트랜잭션 경계를 어떻게 가져가냐입니다. 예를 들어 룰 판정 결과를 DB에 저장하는 로직과 외부 차단 API 호출은 트랜잭션을 분리하는 게 좋습니다. 외부 API 실패로 판정 이력 저장까지 rollback되면 감사 추적이 깨지기 때문입니다. 이처럼 어떤 예외를 비즈니스 실패로 볼지, 어떤 예외를 시스템 실패로 볼지 구분해서 rollback 정책을 명확히 잡는 것이 중요합니다.

#### B-3. 리스크 판정 로직에서 DB 트랜잭션 경계는 어디에 잡는 것이 좋을까요?

외부 차단 API 호출이나 Kafka 이벤트 발행을 DB 트랜잭션 안에 오래 묶는 것은 피하는 편이 좋다고 생각합니다. 먼저 거래 이벤트와 룰 판정 결과를 내부 DB에 저장하고, 이후 차단 또는 알림 처리를 별도로 수행하는 구조가 안전합니다. 판정 결과를 알 수 없는 경우에는 `PENDING_REVIEW` 같은 중간 상태로 저장해 운영자가 수동으로 확인하거나 배치 재처리가 가능하게 해야 합니다.

#### B-4. 예외 처리는 어떤 기준으로 설계하나요?

비즈니스 예외와 시스템 예외를 구분합니다. 예를 들어 이미 처리된 거래, 중복 룰 적용 요청, 정책 임계치 초과는 비즈니스 예외로 명확한 코드와 메시지를 내려야 하고, DB 장애나 외부 API timeout은 시스템 예외로 로그와 알림, 재처리 대상이 되어야 합니다. 운영 관점에서는 에러 코드, 거래번호, 룰 ID, trace id를 함께 남겨 원인 추적이 가능하게 하는 것이 중요합니다.

#### B-5. JPA와 MyBatis 또는 SQL 직접 사용의 장단점은 무엇인가요?

JPA는 도메인 모델 중심 개발과 변경 감지, 연관관계 관리에 장점이 있지만 복잡한 조회나 대량 배치에서는 실행 SQL을 명확히 확인해야 합니다. MyBatis나 SQL 직접 사용은 복잡한 리스크 집계 쿼리, 튜닝이 필요한 통계, 대량 처리에서 제어력이 좋습니다. Qoo10에서 정산 Stored Procedure와 실행계획 튜닝을 했던 경험상, Risk Management처럼 성능과 정확성이 중요한 영역에서는 ORM을 쓰더라도 핵심 쿼리는 실행계획과 인덱스를 반드시 확인해야 한다고 생각합니다.

#### B-6. API 설계 시 중요하게 보는 기준은 무엇인가요?

요청/응답 스펙의 명확성, 상태 전이의 일관성, 실패 시 재시도 가능성, 로그 추적 가능성을 중요하게 봅니다. Risk API는 같은 거래 이벤트가 두 번 들어와도 결과가 예측 가능해야 하므로 idempotency key나 거래번호 기반 중복 판정 방지가 필요합니다. 또한 응답에는 룰 판정 결과, 적용된 룰 ID, 판정 근거를 명확히 포함해야 운영자와 외부 시스템이 결과를 신뢰할 수 있습니다.

#### B-7. 인증/인가는 어떻게 이해하고 있나요?

인증은 사용자가 누구인지 확인하는 과정이고, 인가는 인증된 사용자가 특정 리소스나 기능에 접근할 권한이 있는지 확인하는 과정입니다. Risk Management 시스템에서는 내부 서비스 간 인증, API key/secret 관리, IP 제한, 서명 검증 같은 서버 간 인증이 중요합니다. 또한 룰 정책 변경이나 차단 이력 조회는 접근 권한을 엄격히 관리해야 하고, 감사 로그에 접근자 정보도 남겨야 합니다.

#### B-8. 동시에 같은 거래 이벤트가 두 번 들어오면 어떻게 막을 수 있나요?

우선 거래번호나 이벤트 ID에 unique constraint를 걸어 DB 레벨에서 중복을 막습니다. 애플리케이션에서는 idempotency key를 받아 최초 요청의 판정 결과를 저장하고, 동일 키 요청은 새로 처리하지 않고 기존 결과를 반환하도록 설계할 수 있습니다. Kafka at-least-once 환경에서는 Consumer에서 거래 ID 기반 중복 체크를 거치거나, DB unique constraint가 최종 방어선이 됩니다.

#### B-9. 배치 처리 경험과 실시간 처리의 차이를 설명해 주세요.

Risk Management 시스템은 두 가지를 모두 필요로 합니다. 실시간 처리는 Kafka로 유입된 거래 이벤트를 즉시 소비해 룰 판정을 내리고 차단 여부를 결정하는 흐름으로, 지연 시간과 장애 격리가 핵심입니다. 배치는 일정 기간 거래 데이터를 모아 일별/주별 리스크 집계, 이상 패턴 리포트 생성, 규제 대응 데이터 산출에 사용하며, 처리량, 재시작 가능성, 실패 건 분리, 정합성 검증이 중요합니다. 저는 Qoo10에서 대용량 정산 배치 튜닝을 했고, Kafka 기반 실시간 로그 파이프라인도 운영했기 때문에 두 관점 모두 실무에서 경험했습니다.

### C. 결제 Risk Management 시스템

#### C-1. FDS(이상 거래 탐지 시스템)가 어떻게 동작하는지 아는 대로 설명해 주세요.

FDS는 결제 시도 시점에 사용자 행동, 거래 패턴, 디바이스 정보, 금액/횟수 임계치 등 다양한 신호를 기반으로 이상 여부를 판단하는 시스템입니다. 크게 룰 기반 탐지와 모델 기반 탐지로 나뉘는데, 백엔드 관점에서는 Kafka로 유입된 거래 이벤트를 소비해 룰 엔진에서 임계치/패턴 조건을 평가하고, 판정 결과를 RDB에 저장하고 고위험 건은 차단 또는 운영자 검토 큐로 분기하는 흐름으로 이해하고 있습니다. 판정 이력은 감사 추적과 규제 대응을 위해 전량 보관해야 합니다.

#### C-2. 리스크 룰 엔진을 어떻게 설계하겠습니까?

룰을 코드에 하드코딩하지 않고 DB나 설정 저장소에서 동적으로 로드하는 구조가 중요합니다. 룰은 조건(금액 > X, 단위 시간 내 횟수 > Y 등)과 판정 결과(차단/검토/허용), 우선순위로 구성하고, Service에서 거래 이벤트를 받아 룰 목록을 순회하며 매칭 여부를 평가합니다. 룰 변경 시 재배포 없이 적용할 수 있도록 캐시(Redis 또는 로컬 캐시)에 룰을 유지하고 변경 시 무효화하는 패턴을 쓸 수 있습니다. 판정에 사용된 룰 ID와 매칭 근거는 반드시 이력으로 저장해야 재현과 감사가 가능합니다.

#### C-3. Velocity Check란 무엇이고 어떻게 구현하겠습니까?

Velocity Check는 단위 시간 내 특정 사용자/IP/카드 등이 과도하게 거래를 시도하는지 감지하는 기법입니다. 예를 들어 동일 사용자가 1분 내 5회 이상 결제를 시도하거나, 동일 카드로 10분 내 누적 금액이 일정 이상이면 이상 거래로 판단하는 방식입니다. 구현은 Redis의 `INCR` + `EXPIRE`를 활용해 `user:{id}:count:{window}` 키로 카운팅하고 임계치를 초과하면 차단 판정을 내리는 방식이 간단하고 효과적입니다. 윈도우 기반 집계가 정교하게 필요하다면 Redis Sorted Set을 활용한 슬라이딩 윈도우 방식도 고려할 수 있습니다.

#### C-4. 탐지 룰을 배포 없이 동적으로 변경할 수 있는 구조는 어떻게 만들겠습니까?

룰 정의를 DB 테이블로 관리하고, 애플리케이션 시작 시 로드해 로컬 캐시에 유지합니다. 관리자 API를 통해 룰을 수정하면 DB에 반영하고, 동시에 Redis pub/sub 또는 메시지 큐로 캐시 무효화 이벤트를 발행해 각 서버 인스턴스가 최신 룰을 다시 로드하도록 합니다. 변경 이력은 별도 테이블에 남겨 언제 누가 어떤 룰을 바꿨는지 추적 가능하게 하고, 롤백도 이력 기반으로 가능하게 설계하는 것이 중요합니다.

#### C-5. Kafka 기반 실시간 거래 이벤트를 소비해 룰을 적용하는 흐름을 설명해 주세요.

결제 시스템에서 거래 이벤트가 발생하면 Kafka 토픽에 발행하고, Risk Consumer가 이를 소비해 룰 엔진에 거래 정보를 전달합니다. 룰 엔진은 현재 룰 목록을 평가해 판정을 내리고, 결과를 RDB에 저장합니다. 고위험 판정은 차단 API 호출이나 운영자 알림으로 분기하고, 정상 판정은 후속 처리로 흘려보냅니다. Consumer는 at-least-once이므로 거래번호 기반 멱등성 처리가 필요하고, 처리 실패 건은 DLQ(Dead Letter Queue)로 분리해 재처리할 수 있어야 합니다.

#### C-6. 이상 탐지 결과를 자동 차단과 수동 검토로 나누는 기준은 무엇인가요?

일반적으로 리스크 스코어가 명확하게 임계치를 넘는 고위험 건은 자동 차단, 중간 수준의 의심 건은 수동 검토 큐로 분기하는 방식을 씁니다. 자동 차단 룰은 False Positive가 낮은 패턴에만 적용하고, 신규 룰이나 복잡한 패턴 판정은 초기에 수동 검토 비율을 높여 정확도를 검증한 뒤 자동 차단으로 전환하는 방식이 안전합니다. 수동 검토 케이스는 운영자 화면에서 이력, 판정 근거, 유사 거래를 함께 볼 수 있어야 판단이 빠릅니다.

#### C-7. AML(자금세탁방지) 또는 전자금융거래법 대응을 위한 로그/감사 추적은 어떻게 설계하겠습니까?

거래 발생부터 룰 판정, 차단/허용 결정, 운영자 처리까지 모든 단계를 불변 이력으로 저장해야 합니다. 각 이력에는 거래번호, 사용자/가맹점 ID, 금액, 판정 시각, 적용 룰 ID, 판정 결과, 처리자 정보를 포함해야 합니다. 민감 정보는 마스킹 또는 암호화하고, 이력 테이블은 수정/삭제를 제한해 무결성을 유지해야 합니다. 규제 보고 요건이 있다면 특정 기간의 이력을 빠르게 추출할 수 있는 인덱스 설계와 배치 리포팅 기능도 필요합니다.

#### C-8. 결제 Risk 관련 모니터링에는 어떤 지표를 봐야 하나요?

이상 거래 탐지 건수, 자동 차단 건수, 수동 검토 건수, False Positive/Negative 비율, 룰별 매칭 건수, Consumer 처리 지연, DLQ 누적 건수, 룰 판정 응답 시간 등을 중요하게 봐야 한다고 생각합니다. 저는 Elasticsearch/Grafana/Kibana 기반 모니터링과 Telegram 알림 체계를 구축한 경험이 있어, Risk 지표도 대시보드로 시각화하고 임계치 기반 알림을 연동하는 구조를 만들 수 있습니다.

### D. 데이터베이스 / 대용량 처리

#### D-1. 인덱스 설계 시 어떤 기준을 보나요?

쿼리의 where 조건, join 조건, order by/group by, 카디널리티, 조회 빈도와 쓰기 비용을 함께 봅니다. Qoo10에서 정산 SP 튜닝을 할 때 실행계획을 보고 Full Scan, 비효율적인 Join/Subquery, Cursor 구간을 줄였고, 조회 패턴에 맞춰 Non-Clustered/Composite Index를 설계했습니다. Risk Management에서는 거래번호, 사용자 ID, 판정 결과, 거래 일시 같은 컬럼이 주요 조회 조건이 될 가능성이 높아 복합 인덱스 설계가 중요할 것 같습니다.

#### D-2. SQL 튜닝을 어떻게 진행했나요?

먼저 느린 쿼리와 실행계획을 확인해 병목이 Full Scan인지, Join 순서인지, 인덱스 미사용인지, 불필요한 Cursor/반복 호출인지 파악합니다. 이후 조건절과 인덱스 구조를 맞추고, 필요한 경우 집합 기반 처리로 바꾸며, 변경 전후 실행 시간과 IO를 비교합니다. Qoo10에서는 정산 Stored Procedure 처리 시간을 수십 초에서 수 초 수준으로 줄인 경험이 있습니다.

#### D-3. Lock과 Deadlock을 설명해 주세요.

Lock은 동시에 같은 데이터를 변경할 때 정합성을 지키기 위한 제어 장치입니다. Deadlock은 두 트랜잭션이 서로 상대가 가진 Lock을 기다리면서 더 이상 진행하지 못하는 상태입니다. 예방하려면 트랜잭션을 짧게 유지하고, 테이블/레코드 접근 순서를 일관되게 가져가며, 적절한 인덱스로 불필요한 범위 Lock을 줄이는 것이 중요합니다. Risk Management에서는 동일 거래에 대한 룰 판정이 동시에 여러 Consumer에서 처리되지 않도록 상태 전이와 Lock 범위를 신중히 설계해야 합니다.

#### D-4. 트랜잭션 격리 수준을 설명해 주세요.

격리 수준은 동시에 실행되는 트랜잭션 간 데이터가 어떻게 보이는지를 정하는 기준입니다. Read Uncommitted, Read Committed, Repeatable Read, Serializable 순으로 격리가 강해지고 동시성은 낮아질 수 있습니다. 격리 수준에 따라 dirty read, non-repeatable read, phantom read 문제가 달라집니다. Risk Management에서는 무조건 높은 격리 수준을 쓰기보다, unique constraint, 상태 조건부 update, lock을 조합해 필요한 정합성을 확보하는 것이 현실적이라고 생각합니다.

#### D-5. 대용량 데이터 처리 경험을 설명해 주세요.

위메프/큐텐에서 상품 검색 색인 파이프라인을 운영하며 대량 상품 데이터의 Bulk Indexing, Refresh, Alias 전환, 증분 색인 정책을 다뤘습니다. Kafka 기반 통합 로그 파이프라인도 설계해 API 로그를 Kafka로 수집하고 Elasticsearch에 저장해 실시간 분석 환경을 만들었습니다. Qoo10에서는 정산 관련 대용량 조회와 Stored Procedure 튜닝을 했습니다. 이 경험은 Risk Management의 대용량 거래 로그 파이프라인이나 리스크 집계 배치에도 연결된다고 생각합니다.

#### D-6. 리스크 분석 배치를 설계한다면 어떤 점을 고려하나요?

집계 기준일, 사용자/가맹점/결제수단 단위 리스크 지표, 룰별 탐지 건수/차단 건수를 명확히 정의해야 합니다. 대용량 처리를 위해 배치 단위와 chunk 크기, 재시작 지점, 실패 건 분리, 중복 실행 방지, 결과 검증 리포트가 필요합니다. 또한 배치 결과를 운영자 대시보드나 규제 보고 시스템에 연결할 수 있어야 하고, 비정상 집계 결과는 알림으로 운영자에게 전달되어야 합니다.

#### D-7. Redis나 Kafka를 Risk Management 시스템에서 어떻게 활용할 수 있을까요?

Redis는 Velocity Check를 위한 단위 시간 내 카운팅, 룰 캐시 유지와 무효화, 이상 탐지 결과의 빠른 조회에 활용할 수 있습니다. 다만 리스크 판정 이력의 최종 기준은 RDB가 되어야 하고 Redis는 보조 수단으로 보는 것이 안전합니다. Kafka는 결제 이벤트를 Risk Consumer가 구독해 실시간으로 룰을 적용하고, 판정 결과를 후속 시스템(알림, 차단 API, 운영자 검토 큐)으로 비동기 분기하는 데 활용할 수 있습니다. 저는 Kafka 토픽/파티션 설계와 Redis Replication/Sentinel, Slowlog/Latency 모니터링 경험이 있습니다.

### E. Elasticsearch / Kafka / Redis / 운영

#### E-1. Elasticsearch 운영 경험을 설명해 주세요.

알바천국과 위메프/큐텐에서 Elasticsearch 클러스터를 운영했습니다. Risk Management 관점에서 연결되는 경험은 크게 두 가지입니다. 첫째, 대용량 데이터의 실시간 색인과 집계입니다. Bulk Indexing, Alias 전환, 증분 색인 정책을 다뤘고, 이는 거래 이벤트를 실시간으로 색인하고 이상 패턴을 Aggregation으로 집계하는 구조와 동일합니다. 둘째, 클러스터 운영 안정성입니다. Metricbeat 기반 메트릭 수집과 Grafana/Kibana 대시보드, Telegram 알림을 구성해 장애를 조기 감지했습니다. 검색 쪽에서는 Query DSL, Mapping, Analyzer, nori, edge_ngram 등을 다뤘지만, Risk에 직접 연결되는 건 집계·시각화·운영 안정성 경험이라고 생각합니다.

#### E-2. Elasticsearch 장애 대응 경험은 무엇이 있나요?

검색 결과 누락, 매핑 충돌, 색인 지연, 리소스 사용량 증가 같은 이슈를 다뤘습니다. 장애가 발생하면 먼저 Cluster Health, shard 상태, GC, query cache, disk, CPU/memory를 확인하고, 색인 로그와 검색 로그를 함께 확인했습니다. 알바천국에서는 Rust 기반 로그/메트릭 수집과 Telegram 알림을 구축해 장애 대응 시간을 30분에서 10분 수준으로 줄였습니다.

#### E-3. 검색 템플릿 배포 자동화는 왜 만들었고 효과는 무엇이었나요?

기존 Mustache Template 검색 쿼리는 수작업 관리라 배포 충돌, 변경 이력 부재, 책임자 추적 문제가 있었습니다. 그래서 Rust CLI로 Git Repository의 템플릿을 파싱하고 Elasticsearch `_scripts` API로 자동 등록/수정하는 Jenkins 파이프라인을 만들었습니다. 결과적으로 배포 시간이 0~1분 수준으로 줄었고, 개발/운영 브랜치를 분리해 운영 리스크와 충돌을 줄였습니다. 이 경험에서 배운 핵심은 "운영 중인 시스템에서 설정/정책을 바꿀 때는 변경 이력과 검증 단계가 없으면 장애 원인 추적이 어려워진다"는 점입니다. Risk Management에서 룰 정책을 변경할 때도 동일한 문제가 발생합니다. 누가 어떤 룰을 언제 바꿨는지 추적 가능해야 하고, 잘못된 룰 변경이 이상 탐지 정확도에 영향을 주지 않도록 변경 검증 단계가 있어야 합니다. 그 자동화 경험을 룰 배포 자동화와 이력 관리에 그대로 적용할 수 있다고 봅니다.

#### E-4. Kafka 기반 로그 파이프라인을 어떻게 설계했나요?

위메프/큐텐에서 API -> Kafka -> Elasticsearch 구조의 통합 파이프라인을 설계했습니다. 이 경험에서 Risk Management 거래 이벤트 파이프라인과 직접 연결되는 포인트가 몇 가지 있습니다. 첫째, 토픽 설계입니다. 당시 그룹사/로그 성격별로 토픽을 분리했는데, 거래 이벤트라면 결제 수단별·이상 판정 결과별로 토픽을 나눠 Consumer가 독립적으로 처리할 수 있게 해야 합니다. 둘째, Consumer 안정성입니다. at-least-once 보장 환경에서 처리 실패가 발생하면 DLQ로 분리해 수동 재처리 또는 자동 재시도 구조를 만들었는데, 거래 이벤트 파이프라인에서도 동일하게 필요한 구조입니다. 셋째, 처리 성능입니다. Kafka + Logstash 구조를 Kafka + Vector로 전환해 메모리와 처리 속도를 개선했던 경험처럼, 거래량이 몰리는 시간대에 Consumer 처리 지연이 발생하지 않도록 설계하는 것이 중요합니다.

#### E-5. Redis 운영 시 주의한 점은 무엇인가요?

Redis는 빠르지만 싱글 스레드 기반이므로 O(N) 계열 고비용 명령어를 조심해야 합니다. 위메프에서 Redis master-slave 구조와 모니터링을 구축했고, `rename-command`로 위험 명령어를 제한했습니다. Slowlog와 Latency를 수집해 성능 저하를 조기 감지하는 체계도 만들었습니다. Risk Management에서 Redis를 Velocity Check나 룰 캐시에 쓴다면, Redis 장애 시에도 RDB 기반 폴백 판정이 가능하도록 설계해야 한다고 생각합니다.

#### E-6. 모니터링/알림 시스템을 어떻게 구축했나요?

Metricbeat, Heartbeat, Rust 기반 수집 로직, Elasticsearch, Grafana/Kibana, Telegram 알림을 조합해 Elasticsearch/Kafka/Redis 상태를 모니터링했습니다. CPU, memory, disk, shard, GC, query cache, Kafka 처리량, Redis memory/latency 같은 지표를 수집했고 임계치 기반 알림을 구성했습니다. Risk Management 시스템에서는 이상 탐지 건수, 룰별 차단 비율, Consumer 처리 지연, DLQ 누적, 판정 응답 시간 같은 지표가 중요할 것 같습니다.

#### E-7. Linux 마이그레이션에서 가장 중요했던 점은 무엇인가요?

데이터 손실 없이 기능 호환성을 검증하는 것이 가장 중요했습니다. Windows 클러스터의 설정, 템플릿, Analyzer, Mapping, 색인 데이터를 분석하고 Linux 클러스터에 이관한 뒤 정적/증분 색인을 비교했습니다. systemd 기반 서비스 관리와 로그 운영 체계도 재구성했습니다. Risk Management 시스템에서도 인프라 전환이나 레거시 개선 시 기존 거래 이력과 룰 판정 이력의 정합성 검증이 핵심이라고 생각합니다.

### F. 인성 / 협업 / 장애 대응

#### F-1. 장애를 해결했던 경험을 말해 주세요.

Elasticsearch 클러스터 운영 중 검색 장애나 품질 이슈가 발생하면 쿼리, 사전, Analyzer, 색인 상태, 노드 리소스를 함께 확인해 원인을 좁혔습니다. 또한 장애 대응 시간을 줄이기 위해 로그와 메트릭을 자동 수집하고 Telegram 알림을 붙였습니다. 단순히 장애를 한 번 해결하는 데서 끝내지 않고, 같은 문제가 다시 발생했을 때 더 빨리 감지하고 대응할 수 있도록 모니터링 체계를 만든 점이 의미 있었습니다.

#### F-2. 운영 중 실수를 방지하기 위해 어떤 장치를 두나요?

수작업을 줄이고 자동화와 검증 단계를 둡니다. 검색 템플릿 배포도 수작업으로 운영하면 충돌과 누락이 생길 수 있어 Git/Jenkins 기반 자동 배포로 바꾸고, 개발/운영 브랜치를 분리했습니다. 인프라 설치도 Bash Script로 표준화해 서버별 설정 편차를 줄였습니다. Risk Management에서도 룰 변경 이력 관리, 배포 승인 절차, dry-run, 변경 전후 판정 건수 비교 같은 검증 장치가 필요하다고 생각합니다.

#### F-3. 협업 시 갈등이 있었던 경험과 해결 방법은 무엇인가요?

검색 템플릿 관리 방식처럼 여러 개발자가 함께 사용하는 영역에서는 변경 책임과 배포 타이밍이 불명확해 충돌이 생길 수 있습니다. 이때 특정 사람의 실수로 보기보다 프로세스 문제로 보고, Git 기반 이력 관리와 Jenkins 자동 배포 구조를 제안하고 구현했습니다. Risk Management에서도 룰 정책 변경이나 판정 로직 수정은 여러 팀(운영/준법/개발)의 협의가 필요한 경우가 많아 변경 이력과 승인 흐름이 중요하다고 생각합니다.

#### F-4. 모르는 기술을 빠르게 익힌 경험이 있나요?

위메프 검색개발팀 인력 공백 상황에서 Elasticsearch 기반 상품 검색 플랫폼을 단독으로 인계받은 경험이 있습니다. Query DSL, Analyzer, nori 형태소 분석기, 사전, 색인 파이프라인, 클러스터 운영까지 빠르게 파악해야 했습니다. 운영 이슈를 해결하면서 공식 문서와 기존 코드, 로그를 함께 보며 학습했고, 이후 검색 품질과 운영 안정성을 개선했습니다. Risk Management 도메인도 처음부터 겸손하게 배우되, 거래 로그와 룰 판정 이력을 빠르게 파악하는 방식으로 적응하겠습니다.

#### F-5. 주도적으로 개선한 경험을 말해 주세요.

Elasticsearch 검색 템플릿 배포 자동화 시스템을 주도적으로 구축했습니다. 기존에는 수작업 배포라 시간이 오래 걸리고 변경 이력 추적이 어려웠습니다. Rust CLI와 Jenkins Pipeline, Git 브랜치 전략을 도입해 템플릿 등록/수정을 자동화했고, 배포 시간을 90% 이상 단축했습니다. Risk Management에서도 룰 배포 자동화, 판정 이력 대시보드, 이상 탐지 알림 체계처럼 운영 효율과 신뢰성을 높이는 개선을 주도하고 싶습니다.

#### F-6. 본인이 생각하는 좋은 백엔드 개발자는 어떤 사람인가요?

기능을 구현하는 데서 끝나지 않고, 운영 중 어떤 문제가 생길 수 있는지까지 생각하는 사람이라고 봅니다. 특히 Risk Management처럼 중요한 시스템에서는 정상 케이스보다 오탐, 미탐, 이벤트 유실, 룰 변경 오적용, 판정 이력 누락을 더 꼼꼼히 봐야 합니다. 저는 검색과 데이터 인프라를 운영하면서 장애를 빨리 감지하고, 원인을 추적하고, 재발을 줄이는 구조를 만드는 일을 많이 해왔습니다. 그런 관점을 가진 백엔드 개발자로 기여하고 싶습니다.

## 5. 최종 면접 준비 요약

### 5-1. 면접에서 강조해야 할 핵심 강점 3가지

1. 실시간 데이터 파이프라인 및 이상 탐지 운영 경험  
   Kafka 기반 로그 파이프라인, Elasticsearch 기반 집계/시각화, Grafana/Telegram 모니터링 알림 체계를 직접 구축한 경험이 Risk Management 시스템의 거래 이벤트 파이프라인 및 이상 탐지 모니터링과 직접 연결됩니다.

2. 데이터 정합성과 대용량 처리 경험  
   정산 Stored Procedure 튜닝, 대량 색인 파이프라인, Kafka 로그 파이프라인, 개인정보 파기 스케줄러 경험이 리스크 집계 배치 및 감사 로그 관리와 연결됩니다.

3. 레거시 분석 및 구조 개선 역량  
   Mustache Template -> Query DSL 전환, Windows -> Linux 마이그레이션, Logstash -> Vector 전환처럼 기존 구조를 분석하고 안정적으로 개선한 경험이 있습니다.

### 5-2. 다날 결제 Risk Management 개발 BE와 연결되는 포인트

- 다날은 PG/통합결제 플랫폼 기업이고, Risk Management 개발은 이상 거래 탐지/룰 엔진/모니터링/감사 추적이 중요합니다.
- 본인의 Kafka 기반 실시간 로그 파이프라인 경험은 거래 이벤트 스트림 소비와 룰 적용 흐름에 직접 연결됩니다.
- Elasticsearch 기반 로그 분석/집계/시각화 경험은 거래 이상 패턴 탐지 및 Kibana 대시보드 운영과 연결됩니다.
- Redis/Kafka/DB 운영 경험은 Velocity Check, 룰 캐시, 비동기 이벤트 처리 설계에 연결됩니다.
- Qoo10 정산 SP 튜닝 경험은 대용량 리스크 집계 배치 및 감사 데이터 조회 최적화에 연결됩니다.
- 모니터링/알림 구축 경험은 이상 탐지 건수, 차단 비율, Consumer 처리 지연 같은 Risk 지표 모니터링 역량으로 어필할 수 있습니다.

### 5-3. 면접 전 반드시 복습할 기술 키워드

- Java: Collection, Stream, Exception, Thread 기본
- Spring Boot: MVC 구조, Bean, DI, AOP, Filter/Interceptor
- Transaction: `@Transactional`, propagation, isolation, rollback
- DB: index, execution plan, lock, deadlock, isolation level, unique constraint
- SQL: join, group by, covering index, pagination, batch update
- Spring Batch: Job, Step, Chunk, Reader/Processor/Writer, restart, skip/retry
- Risk Management: FDS, 룰 엔진, Velocity Check, 이상 거래 탐지, 스코어링, 차단/검토 분기
- 안정성: idempotency, retry, timeout, DLQ, 멱등성, 보상 처리
- Kafka: topic, partition, consumer group, offset, at-least-once, DLQ, ordering
- Redis: cache, TTL, INCR/EXPIRE, sliding window, distributed lock, replication, sentinel
- Elasticsearch: index, shard, replica, mapping, analyzer, alias, bulk indexing, aggregation
- 운영/감사: monitoring, alerting, trace id, log correlation, AML, 전자금융거래법, runbook

### 5-4. 답변 시 조심해야 할 부분

- "Risk Management를 잘 안다"고 과장하지 말고, "Kafka 파이프라인/Elasticsearch/모니터링 경험을 Risk Management 도메인에 연결하고 있다"고 말하는 편이 자연스럽습니다.
- Java/Spring 경험은 실제 프로젝트에서 어느 정도까지 했는지 솔직하게 말하고, 대신 데이터 흐름/운영/이상 탐지 관점의 강점을 강조하세요.
- Elasticsearch 경험만 길게 말하면 직무와 멀어질 수 있으니, 항상 "거래 로그 분석, 이상 패턴 집계, 실시간 시각화"로 Risk Management와 연결해 마무리하세요.
- Redis Velocity Check, Kafka 이벤트 소비 같은 기술을 만능처럼 말하지 말고, 최종 판정 이력과 정합성은 RDB와 감사 로그로 유지한다고 답하는 것이 좋습니다.
- 룰 엔진에서 자동 차단이 과도하게 적용되면 정상 거래 차단(False Positive) 위험이 있으므로, 반드시 수동 검토 분기, 룰 정확도 검증 프로세스를 함께 언급하세요.

### 5-5. 마지막 역질문 예시

1. 다날 결제 Risk Management 팀에서는 현재 룰 기반 탐지와 모델 기반 탐지 중 어느 쪽을 더 중심으로 운영하고 있나요?
2. 거래 이벤트를 실시간으로 처리하는 파이프라인에서 현재 팀이 가장 중점을 두는 기술 과제는 무엇인가요?
3. 룰 정책 변경이 발생할 때 개발팀과 운영/준법팀 간 협업 프로세스는 어떻게 이루어지나요?
4. 이상 탐지 결과의 False Positive를 줄이기 위해 현재 어떤 방식으로 룰 정확도를 검증하고 있나요?
5. 입사 후 초반 3개월 동안 기대하는 역할이나 가장 먼저 맡게 될 가능성이 높은 업무는 무엇인가요?
6. 개발자가 운영 안정성 개선이나 자동화 개선을 제안하고 반영할 수 있는 문화가 어느 정도 열려 있는지 궁금합니다.
